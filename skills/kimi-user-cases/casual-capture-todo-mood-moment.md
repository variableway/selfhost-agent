# 随心记录：从碎片消息到结构化人生数据

> 分析日期：2026-04-16

---

## 核心命题

**用户在任何时刻、任何场景下，随手发一条消息给 AI，AI 自动识别、分类、归档，并在后续进行周期性分析——让用户发现自己生活的模式和盲区。**

这也许是所有个人助理场景中**最自然、最无门槛、最能形成长期依赖**的入口。

---

## 为什么"随心记录"是完美的破冰场景？

| 维度 | 说明 |
|------|------|
| **零意图门槛** | 不需要用户"发起一个任务"，只是随手发句话 |
| **零格式门槛** | 可以是文字、图片、语音、链接、文件 |
| **即时反馈** | AI 立即回复"已记录为 Todo/心情/此刻"，给用户确认感 |
| **长期复利** | 记录得越多，周回顾和趋势分析越有价值 |
| **社交安全** | 在群里发也不尴尬，因为看起来像"正常聊天" |

**关键洞察**：
> 大多数人不是不想记录生活，而是**懒得打开专门的 App、选择分类、填写字段**。如果记录的方式就是"发一条消息"，记录量会提升 10 倍以上。

## 三类自动分类：Todo / 心情 / 此刻

OpenClaw 收到用户的任意消息后，首先判断这条消息属于哪一类：

### 1. Todo（待办事项）

**特征**：
- 包含明确的行动意向："要..."、"记得..."、"别忘了..."、"下周..."、"买..."
- 涉及未来的时间点或截止期限
- 可以被勾销或完成

**示例**：
- "下周三记得交电费"
- "要买洗面奶了"
- "周五前把报告发给老板"
- "这个想法不错，以后可以做个小程序"

**AI 处理**：
1. 提取行动主体和时间
2. 评估优先级（基于 deadline 远近、关键词 urgency）
3. 存入 Todo 池
4. 在到期前进行提醒

---

### 2. 心情（情绪状态）

**特征**：
- 表达当下的感受、情绪、身体状况
- 经常包含形容词："累"、"开心"、"焦虑"、"舒服"、"烦躁"
- 与具体行动关联较弱

**示例**：
- "今天好累，不想上班"
- "终于跑完 10 公里了，爽！"
- "又被客户气了，烦"
- "莫名其妙有点低落"

**AI 处理**：
1. 情绪识别（正面/负面/中性，细分情绪标签）
2. 强度打分（1-10）
3. 关联可能的触发因素（基于上下文关键词）
4. 存入情绪日志

---

### 3. 此刻（当下的观察与想法）

**特征**：
- 对当前场景、事物、灵感、观察的描述
- 没有明确的 action item
- 也不是纯粹的情绪表达
- 更像是"这一刻我注意到了什么"

**示例**：
- "这家咖啡店的背景音乐居然是坂本龙一"
- "地铁上看到一个穿汉服的小姐姐，真好看"
- "突然想到，为什么所有的 todo App 都一定要分类呢"
- "窗外的梧桐树开始落叶了"

**AI 处理**：
1. 提取核心观察对象和场景标签
2. 判断是否涉及创意灵感（可转 idea 池）
3. 关联地理位置/时间戳
4. 存入"此刻"流（类似 Instagram Stories 的时间线）

---

## 分类不确定时的智能确认

当 AI 对分类不确定时，可以用**低摩擦的确认方式**，而不是让用户手动选择：

```
用户：又要交房租了，这个月好快

OpenClaw:
┌─────────────────────────────────────────┐
│  📝 已记录："又要交房租了，这个月好快"   │
│                                         │
│  我理解为：                              │
│  [💰 Todo] 交房租（建议设为每月 1 号提醒）│
│  [😔 心情] 感叹时间过得快                 │
│                                         │
│  如果不对，点击修正 👇                   │
│  [这是 Todo] [这是心情] [这是此刻]        │
└─────────────────────────────────────────┘
```

**设计要点**：
- 默认给出一个"最可能的分类"，减少用户操作
- 只提供一键修正按钮，不需要用户打字解释
- 用户的每一次修正都会成为模型微调的样本

## 产品实现的关键技术点

### 1. 消息分类模型

**轻量方案**：基于规则 + LLM 少量样本提示

```python
CLASSIFICATION_PROMPT = """
请判断以下用户消息属于哪一类，并给出置信度（0-1）。

分类选项：
- todo: 包含明确的行动意向、待办事项、计划
- mood: 表达情绪、感受、身体状态
- moment: 对当前场景、事物、灵感的观察记录
- mixed: 同时包含多种类型

输出格式：JSON
{
  "primary": "todo|mood|moment|mixed",
  "confidence": 0.0-1.0,
  "extracted_todos": ["..."] （如果是 todo 或 mixed）,
  "mood_tags": ["..."] （如果是 mood 或 mixed）,
  "moment_tags": ["..."] （如果是 moment 或 mixed）,
  "deadline": "YYYY-MM-DD" （如果有）
}

用户消息：{message}
"""
```

**进阶方案**：用用户的历史修正记录做 few-shot，逐步提升准确率。

### 2. 数据存储结构

```json
{
  "capture_id": "cap_20260416143022",
  "user_id": "xiaoli",

    {

    },
    {

    }
```

### 3. 周期性分析任务

使用定时任务（如 cron / 云函数定时触发）自动执行：
- 每周一 09:00：发送周回顾
- 每月 1 日 09:00：发送月回顾
- 每日 21:00：推送明日 Todo 预览 + 逾期提醒

## 关键挑战与应对

### 挑战 1：隐私与安全感
**问题**：用户担心自己的碎碎念被群友看到，或被平台存储分析。

**应对**：
- 私聊模式优先：敏感记录走私聊通道，群聊只展示低敏感内容
- 本地存储优先：Tauri 桌面端可以将数据保存在本地 SQLite
- 透明数据政策：明确告知"你的数据属于你，可随时导出或删除"

### 挑战 2：分类准确率
**问题**：早期模型可能把"今天好累"误分类为 Todo（"休息"）。

**应对**：
- 高置信度时自动分类，低置信度时主动确认
- 记录用户的每一次修正，定期做 few-shot 优化
- 允许用户设置"个人分类规则"（如"提到'运动'优先归为 Todo"）

### 挑战 3：信息过载
**问题**：记录太多后，周回顾变成数据轰炸，用户懒得看。

**应对**：
- 每次回顾只突出 3 个核心洞察
- 用可视化（图表、关键词云）代替大段文字
- 提供"一键忽略此类分析"的选项

### 挑战 4：从记录到行动的转化断层
**问题**：用户享受记录和分析，但最终还是不行动。

**应对**：
- 在周回顾中直接提供"一键转化为 Todo"的入口
- 对反复提到但未完成的事情，AI 主动询问"需要我帮你拆成小步骤吗？"
- 结合社交压力：允许用户把周回顾分享给好友/伴侣，互相监督
