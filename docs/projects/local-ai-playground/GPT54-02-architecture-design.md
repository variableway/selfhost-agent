# GPT54-02 Local AI Playground - 架构设计文档

## 1. 架构目标
Local AI Playground 需要同时满足以下几个核心目标：
- 面向个人用户，尽量保持轻量级
- 既有可视化界面，又能真正操作本地机器
- 支持从教程、配置到实际运行的完整闭环
- 让用户看到执行过程，而不是只看到结果

因此，架构上不适合只做纯 Web，也不适合只做纯命令行，更适合采用“Web 展示 + 本地端执行”的双形态设计。

## 2. 双端架构设计

### 2.1 Web 端
Web 端主要承担内容承载和能力展示，适合：
- 展示产品介绍与 ROADMAP
- 展示各类云 claw 服务入口
- 提供教程阅读与文档浏览
- 提供组件说明与界面示例

Web 端的限制也需要明确保留：
- 无法直接操作本地机器
- 无法直接执行本地安装脚本
- 更适合作为门户、说明中心和轻量体验入口

### 2.2 本地端
本地端是产品的核心执行形态，负责：
- 直接操作本地机器
- 运行安装脚本与配置脚本
- 管理本地工具和依赖
- 展示终端执行过程
- 为教程中的“点击执行”能力提供底层支撑

本地端需要具备比 Web 端更高的系统访问能力，因此建议作为独立桌面应用存在。

## 3. 推荐技术结构

### 3.1 前端界面层
前端界面负责承载以下内容：
- 配置向导
- 教程阅读器
- 工具中心
- 组件展示区
- 终端输出窗口

推荐采用组件化前端框架，重点是保证：
- 页面结构清晰
- 引导过程简单
- 教程和工具能够复用同一套 UI 体系

### 3.2 本地执行层
本地执行层负责把用户在界面中的操作转化成真实的系统行为，例如：
- 安装 Node.js、Python 或其他依赖
- 写入配置文件
- 调用本地 CLI 工具
- 启动浏览器自动化任务

这一层需要统一封装命令执行、日志回传、错误处理和权限提示。

### 3.3 内容管理层
由于产品强调“文档先行”和“教程驱动”，因此需要一个独立的内容管理层来组织：
- 新手教程
- 工具说明
- 常见问题
- 组件说明
- 路线图内容

内容建议优先采用 Markdown 或兼容格式，便于维护、版本管理和后续扩展。

## 4. 核心模块拆分

### 4.1 安装与配置向导模块
该模块负责：
- 选择要安装的工具
- 输入 API Key 和关键配置
- 生成对应配置文件
- 驱动一键安装流程

这是最核心的 MVP 模块，因为它直接承接用户“我想先搭起来”的第一诉求。

### 4.2 教程与学习模块
该模块负责：
- 用分步形式展示教程
- 解释每一步的作用
- 支持边看边操作
- 作为用户从小白到入门的学习路径承载层

### 4.3 工具中心模块
该模块负责集中管理可直接使用的工具，例如：
- AI 助手
- AI 翻译
- AI 图片生成
- 后续扩展的更多轻量效率工具

### 4.4 组件展示模块
该模块负责让用户理解：
- 有哪些 UI Layout 可以复用
- 有哪些 ChatDashboard 类组件可以参考
- 这些组件组合起来可以做出什么样的 AI 工具

它的价值不在于立即生成复杂应用，而在于建立用户对“AI 工具是如何组成的”的直觉。

### 4.5 浏览器自动化扩展模块
该模块是中后期增强能力，负责支持：
- 通过 CLI 驱动 browser
- 结合自动化流程完成日常效率任务
- 为更复杂的 AI 工作流提供扩展接口

## 5. 终端透明化机制
终端透明化是本产品区别于普通教程网站的重要能力。

建议的执行流程如下：
1. 用户在 UI 中点击某个安装或运行操作
2. 本地端接收指令并启动对应命令
3. 后台实时捕获 stdout 和 stderr
4. UI 中同步展示 terminal 输出内容
5. 用户可以看到每一步执行情况、成功提示和报错信息

这样做的价值在于：
- 降低黑盒感
- 提高用户信任
- 让用户逐步理解底层工具链
- 出错时更容易定位问题

## 6. 架构原则
为了让架构真正服务于目标用户，建议坚持以下原则：
- 本地执行能力必须强，但界面必须足够简单
- 教程、工具和组件说明要共用一套内容体系
- 所有关键操作都尽量可回显、可解释、可重试
- MVP 阶段先解决“装得起来、看得懂、用得上”
- 后续再逐步扩展到自动化和自定义能力
